Selling Flow — Design & Implementation Notes

PickupStep + SellStep + dynasty gold deposit (May 2026)

DESIGN NOTE · SELLING FLOW

1. Purpose

Workshops accumulate crafted goods via the crafting flow. Once a stack is full (quantity ≥ stackLimit), a Sell task moves that full stack to the nearest marketplace in exchange for gold, which is then deposited into the owning dynasty's treasury.

  • Trigger: any craftable item in the workshop inventory hits stackLimit.
  • Quantity: one full stack per task.
  • Price: pure item baseValue from the DB (no markup / demand yet).
  • Buyer: nearest entity with MarketplaceComponent (Manhattan distance).
  • Payee: the workshop's owning dynasty entity (NOT the workshop itself).
  • Worker: one employee per work-slot per spawn interval.

2. Architecture Overview

+----------------------+
|  WorkshopSystems     |  PostWorkshopTasks per worker slot:
|  (Alchemy/Carpenter) |    1. Sell phase  → if full stack, post Sell task
|                      |    2. Craft phase → if materials available, post Craft
|                      |    3. Gather phase → fallback
+----------+-----------+
           |
           v
    Sell task posted on workshop's NoticeboardComponent
           |
           v
    TaskClaimSystem assigns to an idle worker of that workshop
           |
           v
    6-step execution (see section 4)
           |
           v
    Gold ends up in dynastyEntity.InventoryComponent

Step executors used:

  • MoveToStepSystem (priority 17) — 3x walks per task
  • PickupStepSystem (priority 19) — take items from workshop
  • SellStepSystem (priority 19) — atomic exchange at marketplace
  • DeliverStepSystem (priority 19) — deposit gold at dynasty

3. Gold as a Database Item

Gold is just another item, defined in data/database/items/currency/gold/desc.json. It plays by all the same rules as any other item: lives in InventoryComponent stacks, transferred by DeliverStep / PickupStep.

Helpers in ItemUtils.h:

  • AddGold(r, entity, amount) — append to entity's inventory
  • GetGoldAmount(r, entity) — total quantity held
  • TryDeductGold(r, entity, amount) — deduct, but skip for UnlimitedGold

UnlimitedGoldComponent is a tag. Any entity carrying it is treated as having infinite gold by TryDeductGold (returns true without touching inventory). Used by MarketplaceSystem — they always pay.

4. Task Structure (6 Steps)

StepSystemPriorityPurpose
0MoveToStepSystem17Walk to workshop tile (in case worker isn't there).
1PickupStepSystem19Pickup stackLimit items from workshop inventory.
2MoveToStepSystem17Walk to the marketplace tile.
3SellStepSystem19Atomic exchange: items worker→buyer, gold buyer→worker.
4MoveToStepSystem17Walk back to the workshop tile.
5DeliverStepSystem19Deposit all gold from the worker into the workshop's owning DYNASTY inventory (GameEntity target).

Step 5 is the subtle one: the worker walks back to the workshop tile (because the dynasty entity has no map position), but the gold deposit targets the dynasty entity itself, not the workshop. The workshop is the dynasty's "local office" — gold goes into the dynasty's abstract treasury.

5. Step Component Structures

5.1 PickupStep

struct PickupStep {
    entt::entity source;            // runtime handle
    int      sourceObjectId;        // Tiled GUID (workshop) - or
    PackedId sourceGameId;          // GameEntity id_index — exactly one is non-zero
    StringId itemType;
    int      quantity;              // 0 = all of itemType
};

The GameEntity source path (sourceGameId) is provided for symmetry with DeliverStep but has NO PRODUCER yet.

5.2 SellStep

struct SellStep {
    entt::entity buyer;             // runtime handle
    int  buyerObjectId;             // Tiled GUID (marketplace)
    StringId itemType;
    int  quantity;                  // >= 1
    int  pricePerUnit;              // cached from item DB baseValue at task creation
};

No GameEntity buyer path yet — marketplaces are always Tiled objects.

5.3 DeliverStep (extended — dual-id)

struct DeliverStep {
    entt::entity target;
    int      targetObjectId;        // Tiled GUID (workshop) - or
    PackedId targetGameId;          // GameEntity id_index (dynasty, city) — one non-zero
    StringId itemType;              // empty = all items
    int      quantity;              // 0 = all of itemType
};

Mirrors the Option-B dual-field pattern already used by TaskComponent.postedBy.

6. Atomicity of SellStep

SellStepSystem is one-shot (gated by stepStarted). In a single tick:

  1. Resolve buyer runtime handle (lazy from buyerObjectId).
  2. Validate buyer + buyer inventory + quantity > 0 + pricePerUnit ≥ 0.
  3. PRE-CHECK: Worker carries quantity of itemType? Buyer can pay (UnlimitedGoldComponent OR gold ≥ totalPrice)? If either fails → task.state = Failed, nothing transferred.
  4. COMMIT (guaranteed to succeed by pre-check):
    • TryDeductGold(buyer, totalPrice) — no-op for UnlimitedGold
    • TryDeductItem(worker, itemType, qty)
    • AddItem(buyer, itemType, qty)
    • AddItem(worker, gold, totalPrice)

The pre-check / commit split keeps the swap atomic — no intermediate state where the worker has lost items but not received gold.

7. Workshop Decision Order

PostWorkshopTasks iterates each free worker slot. Per slot:

  1. Sell phase: For each item in WorkshopCraftingComponent.craftItems: read stackLimit/baseValue from DB. If workshop has stack ≥ stackLimit, find nearest MarketplaceComponent (Manhattan), build 6-step Sell task targeting dynasty inventory, post and break.
  2. Craft phase: any craftable with full materials ≥ 50? Post Craft.
  3. Gather phase: fallback — post Gather+Deliver.

Sell is preferred over Craft because a full stack means capacity is reached — selling before crafting more keeps the workshop responsive.

Backpressure: activeWorkTasks counts Sell tasks the same as Craft/Gather, so a single-worker workshop won't overlap sell trips with other work.

8. Pricing (Current)

pricePerUnit = baseValue from item DB. No markup, no demand curve.

Example: health_potion baseValue=8, stackLimit=10 → one Sell task moves 10 potions for 80 gold.

Future: per-buyer multipliers, item demand, regional pricing.

9. Gold Flow & Dynasty Deposit

Workshop's owning dynasty is resolved from DynastyWorkshopComponent.dynasty at task-post time (NOT at execution time). Captured into DeliverStep:

depositGold.target         = dynastyEntity;
depositGold.targetObjectId = 0;
depositGold.targetGameId   = id_index(dynasty's GameEntityComponent.id);
depositGold.itemType       = DB::items::currency::gold;
depositGold.quantity       = 0;  // all gold the worker carries

After the worker arrives at the workshop tile, the DeliverStep dumps every gold stack it carries into the dynasty's InventoryComponent. DynastySystem Phase C already reads gold via GetGoldAmount(r, dynastyEntity) since the gold-as-item migration, so the dynasty can immediately use the proceeds.

10. Failure Handling

Step-level failure modes:

  • PickupStep fails → task.state = Failed. Source didn't have the items.
  • SellStep fails → task.state = Failed. Worker still holds the items.
  • Worker invalid → task.state = Failed.

On failure the trailing MoveTo + Deliver(gold) steps still exist in the task list, but the task is terminal so they never run. Without recovery, any items the worker picked up earlier would remain stuck in their inventory.

10.1 TaskFailureRecoverySystem (priority 14, Tick)

Watches for Failed Sell tasks via on_destroy<TaskComponent> and posts a follow-up ReturnGoods task on the worker's PERSONAL noticeboard. Two-phase so the signal callback stays lightweight:

Phase 1 (on_destroy callback, fires inside TaskCleanupSystem at pri 20): If task.state == Failed AND task.type == Sell AND claimedBy is valid, push worker entity to pending queue. No registry mutation.

Phase 2 (Update, runs at priority 14 next tick, BEFORE TaskClaim at 15): Drain the queue. For each worker, skip if no inventory or it's empty, skip if no WorkshopWorkerComponent (worker was fired; items stay with them), skip if workshop entity missing pieces. Otherwise, build 2-step ReturnGoods task with postedBy = workerEntity, which routes to the worker's personal NoticeboardComponent (auto-created via get_or_emplace).

10.2 Tick-by-tick walkthrough

  • Tick N: SellStep(19) → task.state = Failed; TaskCleanup(20) → destroys task → on_destroy fires → TaskFailureRecoverySystem captures worker entity.
  • Tick N+1: TaskFailureRecoverySystem.Update(14) → posts ReturnGoods task with postedBy=worker → NoticeboardSystem routes onto worker's personal board; TaskClaimSystem(15) → idle worker walks city + personal + workshop boards, sees ReturnGoods on personal → claims → InProgress; MoveToStep(17) → starts walking back to the workshop tile.
  • Tick N+M: DeliverStep(19) → empties worker's inventory into workshop; Task DONE.

10.3 Policy Decisions

  • Recovery triggers ONLY on Failed Sell tasks. Other Failed task types do not produce recovery tasks.
  • If the worker is fired (no WorkshopWorkerComponent) between failure and recovery, no recovery task is posted.
  • If the worker is fired DURING an in-flight ReturnGoods, the task simply completes — items end up at the workshop they were fired from.

12. ID Usage

  • PickupStep.source — entt::entity, runtime-only
  • PickupStep.sourceObjectId — int (Tiled GUID), used for workshop sources
  • PickupStep.sourceGameId — PackedId (id_index), reserved, no producer yet
  • SellStep.buyer — entt::entity, runtime-only
  • SellStep.buyerObjectId — int (Tiled GUID), marketplaces are Tiled objects
  • DeliverStep.target — entt::entity, runtime-only
  • DeliverStep.targetObjectId — int (Tiled GUID), used for workshop deposits
  • DeliverStep.targetGameId — PackedId (id_index), used for dynasty/city deposits
  • Convention: exactly one of the two ids is non-zero per step instance

13. What Is Not Built Yet

  • In-game way to deliberately fail Sell tasks for verifying recovery (admin/debug button)
  • Smarter sell triggers (queue tasks before stack overflows the limit)
  • Pricing: demand curves, regional, markups, marketplace mark-ups
  • Character-as-seller (independent NPC sells personal goods; sell task omits the final Deliver step)
  • PickupStep from GameEntity sources (sourceGameId branch present but no producer)
  • Save/load reconstruction of step.target / step.source in TilemapWorld::LoadEntities
  • Marketplace inventory cap / restocking (currently accepts items indefinitely, never re-sells)

14. How to Test

  1. Place at least one marketplace and one dynasty-owned workshop on the map. Add a dynasty via the debug panel (seeds 50000 gold).
  2. Watch the workshop's InventoryComponent fill up via the Craft phase. Once any crafted item hits stackLimit (e.g. health_potion ≥ 10), a Sell task is posted next workshop spawn tick.
  3. Watch channel "esc.task" for the 6-step sequence: MoveToStep started → workshop; PickupStep started + done; MoveToStep started → marketplace; SellStep started + done; MoveToStep started → workshop; DeliverStep started + done → target entity; Task DONE.
  4. Verify the dynasty's gold in the debug Dynasties panel — it should increase by quantity × baseValue per completed Sell task.
  5. Verify the marketplace's InventoryComponent accumulates the items it bought.
  6. Trigger gold-driven dynasty behaviour: If dynasty has ≥ MIN_GOLD_TO_BUY (50000), Phase C buys an unowned workshop; if enough gold for upgrade, Phase D upgrades it.
