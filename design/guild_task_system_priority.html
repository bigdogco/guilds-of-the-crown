StepSystemPriorityDescription
0MoveToStepSystem17Walk to the workshop tile.
1CraftStepSystem16Each minute, accumulate progress; on accumulator ≥ 1.0, deduct materials and add the crafted item. Marks task Done when materials run out.

Sell (6 steps)

StepSystemPriorityDescription
0MoveToStepSystem17Walk to the workshop tile.
1PickupStepSystem19Take stackLimit items from workshop → worker inventory.
2MoveToStepSystem17Walk to the marketplace tile.
3SellStepSystem19Atomic exchange: items worker→buyer, gold buyer→worker.
4MoveToStepSystem17Walk back to the workshop tile.
5DeliverStepSystem19Deposit all gold from the worker into the workshop's owning DYNASTY entity (targetGameId path).

ReturnGoods (2 steps) — failure recovery

StepSystemPriorityDescription
0MoveToStepSystem17Walk to the workshop tile.
1DeliverStepSystem19Deliver EVERY stack the worker carries to the workshop (itemType empty + quantity=0). Target uses targetObjectId.

Fallback Task Shapes

  • 2-step — MoveToStep + GatherStep (no workshops present)
  • 1-step — MoveToStep only (no gathering spots present)

Key Design Notes

  • Tag-based step routing (May 2026): each character entity carries exactly one zero-size tag (MoveToStepActiveTag, GatherStepActiveTag, DeliverStepActiveTag, PickupStepActiveTag, SellStepActiveTag, CraftStepActiveTag) while a task is InProgress. Step systems pre-filter via EnTT view intersection for O(N_chars_in_step) instead of O(N_all_chars).
  • Tags are NOT serialised. StepTagRebuildSystem (priority 13) restores them on the first tick after a save load.
  • stepStarted guards one-shot setup (e.g. emplace MoveGoalComponent, trigger transfer). TaskStepsComponent::Advance() resets stepStarted so the next executor picks up immediately on the same tick.
  • TaskClaimSystem (15) runs before PathfindingSystem (16) so a task claimed and set to InProgress in the same tick immediately gets a pathfinding goal.
  • DeliverStepSystem (19) runs after GatherStepSystem (18) so on the tick MoveToStepSystem advances past the walk-to-workshop step, DeliverStep is picked up and completed in the same tick.
  • DynastySystem (19) runs after all step systems so Done hire tasks are still alive for membership finalisation, and before TaskCleanupSystem (20) destroys them.
  • AlchemyShopSystem and CarpenterShopSystem only post work tasks for workshops that carry DynastyWorkshopComponent. Independent workshops stay silent.
  • Both workshop systems live in WorkshopSystems.cpp; BuildingSystems.cpp only carries generic building lifecycle.
  • Workshop systems prefer Sell → Craft → Gather in that order.
  • CraftStepSystem runs at Minute frequency rather than Tick — items are produced on the order of game-hours.
  • Gold is a normal DB item (items/currency/gold). It moves through DeliverStep / PickupStep like anything else. Marketplaces carry UnlimitedGoldComponent.
  • DeliverStep and PickupStep use a dual-id pattern for target/source: a Tiled GUID (*ObjectId) for buildings, OR an id_index of GameEntityComponent.id (*GameId) for runtime GameEntities.
  • Failure recovery for Sell tasks lives in TaskFailureRecoverySystem (pri 14). It uses on_destroy to capture the worker entity, then posts a ReturnGoods task onto the worker's personal noticeboard.

Dynasty Membership Lifecycle

  • DynastySystem posts HireWorker task → TaskClaimSystem (NPC claims it)
  • → MoveToStepSystem walks NPC to workshop → task Done
  • → DynastySystem Phase B emplaces DynastyMemberComponent on NPC.

Firing (dynasty-initiated): DynastySystem::Fire() removes DynastyMemberComponent and updates the dynasty's member lists directly — no task needed.

Components

  • DynastyComponent — on the dynasty entity (name, gold, maxMembers, member/workshop lists).
  • DynastyMemberComponent — on character entities; absent = independent NPC.
  • DynastyWorkshopComponent — on workshop entities; absent = independent workshop.
  • DynastyInitComponent — tag; triggers handle resolution pass in DynastySystem Phase A.
