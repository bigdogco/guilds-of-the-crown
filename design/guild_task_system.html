Task System — Design & Implementation Notes

Multi-step tasks, step executors, gather task (April 2026)

DESIGN NOTE · TASK SYSTEM v1

May 2026 Update

This document captures the original architecture. Extensions since: Hire task type, per-owner NoticeboardComponent, DeliverStep/PickupStep/SellStep, Craft task type, and tag-based step routing. When this doc and newer docs differ, the newer docs win.

1. Purpose

Extend the task system from single-action (MoveTo only) to ordered multi-step sequences. The first real task type is a two-step Gather task:

  • Step 0 — MoveToStep: walk to the gathering spot
  • Step 1 — GatherStep: accumulate items into inventory until target reached

The architecture is designed so that adding future task types (Deliver, Build, Guard, etc.) only requires defining new step structs and a new step executor system — no changes to the claiming, noticeboard, or cleanup logic.

3. Key Concepts

3.1 Task Entities

A task is an entt entity with two components:

  • TaskComponent — shared metadata (id, type, state, claimedBy, timestamps)
  • TaskStepsComponent — ordered step sequence + current step index + stepStarted flag
  • GameEntityComponent — provides a StableId (PackedId) for logging/save

Task entities have no TransformComponent or visual presence. They are pure data describing work to be done.

3.2 Task States

  • Posted — visible on the noticeboard, available for claiming
  • Claimed — reserved by a character, not yet executing
  • InProgress — character is actively working on the current step
  • Done — all steps completed successfully (terminal)
  • Failed — could not complete — path blocked, interrupted, cancelled (terminal)

Terminal states are destroyed by TaskCleanupSystem every tick.

3.3 TaskStepsComponent

struct TaskStepsComponent {
    std::vector<TaskStep> steps;   // all steps, in order
    int  currentStep  = 0;         // index into steps[]
    bool stepStarted  = false;     // reset to false by Advance()
};

stepStarted is the handshake between phase transitions. Each step executor checks (!stepStarted) for first-tick setup. Advance() resets it to false so the next executor picks up on the same tick.

3.4 Step Payload Types

MoveToStep   { destination: Spark::Point, arrivalRadius: float }
GatherStep   { spot: entt::entity, itemType: StringId,
               gatheringSpeed: int, targetQuantity: int,
               gathered: int, accumulator: float }
TaskStep  =  std::variant<MoveToStep, GatherStep, DeliverStep,
                          PickupStep, SellStep, CraftStep>

3.5 Tag-Based Step Routing

Each character entity carries exactly one zero-size tag component while a task is InProgress, identifying its current step type:

  • MoveToStepActiveTag, GatherStepActiveTag, DeliverStepActiveTag
  • PickupStepActiveTag, SellStepActiveTag, CraftStepActiveTag

Step systems pre-filter via EnTT view intersection: reg.view<CurrentTaskComponent, XxxStepActiveTag>()

This is O(N_chars_in_that_step) rather than O(N_all_chars). Tags are NOT serialised — StepTagRebuildSystem (priority 13) rebuilds them on first tick after save load.

5. System Details

5.1 TaskClaimSystem (priority 15)

Owns noticeboard signal registration (on_construct/on_destroy<TaskComponent>).

  • Phase A: Every SPAWN_INTERVAL_TICKS (600), spawns a task if fewer than MAX_ACTIVE_TASKS are alive. Currently always tries Gather if spots are available, falls back to MoveTo.
  • Phase B: Idle characters query the noticeboard for Posted tasks. Each picks the nearest by world-space distance to the first step's destination.
  • Phase C: Kept separate from Phase B so a task can go Posted → Claimed → InProgress in one tick.

5.2 MoveToStepSystem (priority 17)

  • Phase A: Finds InProgress tasks where current step is MoveToStep and !stepStarted. Emplaces MoveGoalComponent, sets stepStarted = true.
  • Phase B: Arrived → Advance() or task.state = Done. Blocked/Cancelled → task.state = Failed.

5.3 GatherStepSystem (priority 18)

No signals needed. Each tick, marks stepStarted on first tick, adds gatheringSpeed / 60.0 to accumulator, converts whole values into inventory items. When gathered ≥ targetQuantity: Advance() or task.state = Done.

5.4 TaskCleanupSystem (priority 20)

  • Phase A: Collects and destroys Done/Failed entities. on_destroy fires automatically, removing from noticeboard.
  • Phase B: Removes stale CurrentTaskComponent, making the character idle.

6. Failure Handling

Failures are terminal — no per-step retry or fallback branch.

Sources: MoveToStep path blocked, external interrupt from another system.

On failure: task.state = Failed → TaskCleanupSystem destroys next tick → CurrentTaskComponent removed → character goes idle → picks new task next tick.

"Drop inventory and flee" is NOT part of the task system — lives in a future ThreatResponseSystem.

7. Adding a New Step Type

  1. Define the step struct in TaskComponents.h
  2. Add it to the TaskStep variant and add STEP_TYPE constant + serialisation
  3. Add a tag and wire it into StepTags.h (declare, add to RemoveActiveStepTag, add branch in ApplyStepTag, add to exclude list in StepTagRebuildSystem)
  4. Create MyStepSystem.h/.cpp — inherit IEntitySystem, priority > 18
  5. Register in GoCApp::Init()
  6. Add the new .cpp/.h to goc.vcxproj

No changes needed to TaskClaimSystem, TaskCleanupSystem, or TaskNoticeboard — they are fully step-type agnostic.

9. StableId Usage

  • Task.id — PackedId via GameEntityComponent; used in all log output, survives save/load
  • Task.claimedBy — entt::entity, runtime-only, rebuilt on claim
  • Task.postedBy — entt::entity, runtime-only for now
  • GatherStep.spot — entt::entity, runtime-only reference to gathering spot
  • CurrentTask.task — entt::entity, runtime-only
  • Noticeboard index — entt::entity handles, rebuilt from registry on load

10. What Is Not Built Yet

  • TaskStepsComponent serialisation (variant + entity refs need custom operators)
  • Scoped noticeboards (per-workshop, per-region, personal needs)
  • Utility scoring for task selection (trait/skill modifiers, weights)
  • Task timeouts or reservation expiry
  • Inventory capacity limit for "fill inventory" completion condition
  • Team tasks (multi-NPC coordination on a single task)
  • UI for decision logs (console logging via "esc.task" channel only)
  • Real task producers (CityAI, NeedsSystem) — WorkshopSystems already posts
  • Interrupt handling via external systems (CombatSystem, ThreatSystem)
  • PickupStep GameEntity source path (sourceGameId branch) needs a real producer

11. How to Test

  1. Build and run with the test world loaded.
  2. The world must have at least one BerryGatheringSpotComponent or WoodGatheringSpotComponent entity for Gather tasks to spawn.
  3. Characters with CharacterComponent + PathfindingComponent + TransformComponent + InventoryComponent will automatically pick up tasks.
  4. Watch console logs (channel "esc.task") for the task lifecycle.
  5. Verify that after a Gather task completes, the character's InventoryComponent contains the expected itemType and quantity.
  6. Verify that failed tasks are cleaned up and characters go idle and pick a new task.
