Task Noticeboard System

Design & Implementation Notes — Updated May 2026

DESIGN NOTE · NOTICEBOARD SYSTEM

1. Purpose

A task system that lets characters autonomously claim and execute multi-step work tasks. Tasks are produced by workshop systems and the dynasty system; a per-owner NoticeboardComponent controls which tasks each board holds, and TaskClaimSystem filters by board membership before claiming.

3. Key Concepts

3.1 Task Entities

A task entity always has exactly two components:

  • TaskComponent — shared metadata (id, state, claimed/posted by, timestamps)
  • TaskStepsComponent — ordered list of TaskStep variants to execute

Task entities also carry GameEntityComponent for a stable PackedId used in logs and save/load. They have no TransformComponent or visual presence.

3.2 Task Types

  • MoveTo — single-step: walk to a tile. Used internally / for testing.
  • Gather — 4-step: walk to spot → gather → walk to workshop → deliver.
  • Hire — 1-step: walk to a dynasty workshop. Arrival = employment.
  • Craft — 2-step: walk to workshop → craft items in place. Posted by workshop systems when materials are stocked above the craft threshold.
  • Sell — 6-step: workshop → pickup → marketplace → sell → workshop → deposit gold into the owning dynasty.
  • ReturnGoods — 2-step: walk to workshop → deliver every item from worker inventory. Posted by TaskFailureRecoverySystem when a Sell task fails, onto the worker's personal noticeboard.

3.3 Task States

  • Posted — visible on the noticeboard, available for claiming
  • Claimed — reserved by a character, not yet executing
  • InProgress — character is actively executing the current step
  • Done — all steps completed successfully (terminal)
  • Failed — could not complete — path blocked, cancelled (terminal)

Terminal states are cleaned up by TaskCleanupSystem (priority 20).

3.4 TaskStepsComponent

struct TaskStepsComponent {
    std::vector<TaskStep> steps;
    int  currentStep  = 0;
    bool stepStarted  = false;
};

Holds a std::vector<TaskStep>, where TaskStep is a std::variant of:

  • MoveToStep — { destination: Spark::Point }
  • GatherStep — { spot, spotId, itemType, gatheringSpeed, targetQuantity, ... }
  • DeliverStep — { target, targetObjectId, targetGameId, itemType, quantity }
  • PickupStep — { source, sourceObjectId, sourceGameId, itemType, quantity }
  • SellStep — { buyer, buyerObjectId, itemType, quantity, pricePerUnit }
  • CraftStep — { itemType, craftHoursPerItem, accumulator, crafted, workshop, workshopObjectId }

DeliverStep and PickupStep use the dual-id pattern: exactly one of *ObjectId (Tiled GUID) or *GameId (id_index of GameEntityComponent.id) is non-zero per step.

3.5 NoticeboardComponent (replaces the old global TaskNoticeboard)

A component — not a global service — attached to any entity that owns tasks.

Current owners:

  • Workshop entity — private board; holds Gather+Deliver work tasks
  • City entity — public board; holds Hire tasks from all workshops
  • (Dynasty / character boards reserved for future use)

Fields: postedTasks — std::vector<entt::entity> — runtime handles to task entities. Runtime-only, not serialised; rebuilt from scratch after load via NoticeboardSystem::RebuildAllBoards().

3.6 NoticeboardSystem (priority 5)

Owns all routing logic — task producers never touch boards directly.

Signal hooks:

  • on_construct<TaskComponent> → OnTaskConstructed: Reads the new task's type, calls ResolveBoardOwner(), adds the task entity to the resolved board's postedTasks list.
  • on_destroy<TaskComponent> → OnTaskDestroyed: Reads postedBy, calls ResolveBoardOwner(), swap-removes the task from the board's list.

Routing rule (ResolveBoardOwner):

  • Hire tasks → city entity (if a city entity exists)
  • All others → task.postedBy (the workshop that posted it)
  • Fallback → task.postedBy (if no city entity exists yet)

postedBy is always the workshop, even for Hire tasks. The board is just where the task is visible.

3.7 City Entity

A singleton entity representing the city. Created at map load time. Carries: CityComponent, GameEntityComponent, NoticeboardComponent, CityInitComponent. CitySystem (priority 6) ensures it always gets GameEntityComponent + NoticeboardComponent.

The city board is the public noticeboard: any character can see tasks on it regardless of dynasty or employment status. Currently holds only Hire tasks.

3.8 CurrentTaskComponent

Attached to a character when it claims a task. Contains:

  • task — entt::entity, runtime handle to the task entity
  • taskId — PackedId, id_index of the task's GameEntityComponent.id (used to re-resolve task entity after save/load)

Characters without CurrentTaskComponent are idle and eligible for claiming. Removed by TaskCleanupSystem after the task entity is destroyed.

4. Task Claiming — Board Visibility and Eligibility

TaskClaimSystem Phase B walks the boards relevant to each character and applies a single employment filter:

Free agent (no WorkshopWorkerComponent):

  • Reads the city's public board (Hire tasks only).
  • All Hire tasks are eligible.
  • Cannot see workshop private boards.

Workshop worker (has WorkshopWorkerComponent):

  • Reads the city board AND their workshop's private board.
  • Hire tasks are skipped (already employed).
  • May claim Gather+Deliver tasks posted by their own workshop.

Board-walking order per character:

  1. City board (public)
  2. Own workshop's board (if WorkshopWorkerComponent present)

For each candidate task the character adds it to a scored set; the nearest (lowest-cost) eligible task is claimed.

5. Task Producers

5.1 AlchemyShopSystem (priority 11) / CarpenterShopSystem (priority 12)

Shared implementation via file-local PostWorkshopTasks template (see WorkshopSystems.cpp).

Per dynasty-owned workshop each spawn interval:

Hire tasks:

  • openHireSlots = maxWorkers - workerCount - pendingHires
  • pendingHires = non-terminal tasks where postedBy == workshopEntity AND type == Hire
  • One Hire task per open slot. postedBy = workshopEntity.
  • NoticeboardSystem routes these to the city's public board automatically.

Work tasks (Sell OR Craft OR Gather+Deliver — one per free worker slot):

  • workTasksToPost = workerCount - activeWorkTasks
  • Decision order per free slot:
    • 1. Sell — any crafted-item stack ≥ stackLimit? Post a 6-step Sell task.
    • 2. Craft — any craftable with full materials ≥ 50? Post Craft.
    • 3. Gather — fallback: post Gather+Deliver.
  • postedBy = workshopEntity; visible only on the workshop's private board.

No MAX_TASKS_PER_WORKSHOP cap — the slot counts above bound task creation naturally.

5.2 DynastySystem (priority 19) — Phase C

For each dynasty with open member slots (maxMembers - members - pendingHires > 0): Post one TaskType::Hire task per open slot, cycling through the dynasty's owned workshops.

6. Step Systems

  • MoveToStepSystem (17): Detects MoveToStep as current step. On stepStarted=false, emplaces MoveGoalComponent. On MoveResultComponent signal: Arrived → Advance(); Blocked/Cancelled → task.state = Failed.
  • GatherStepSystem (18): Detects GatherStep as current step. Each tick, increments accumulator by gatheringSpeed. Whole items added to inventory. When gathered ≥ targetQuantity: Advance() or task.state = Done.
  • DeliverStepSystem (19): Detects DeliverStep as current step. One-shot: transfers all matching items from character inventory to the target's InventoryComponent. Target can be Tiled (targetObjectId) or GameEntity (targetGameId).
  • PickupStepSystem (19): Mirror of DeliverStepSystem — items flow source → worker. Source can be Tiled (sourceObjectId) or GameEntity (sourceGameId — reserved).
  • SellStepSystem (19): Detects SellStep as current step. One-shot atomic exchange at the buyer. Pre-check / commit pattern keeps the swap atomic.
  • TaskFailureRecoverySystem (14): Not a step executor — hooks on_destroy<TaskComponent>. Captures the worker entity from a Failed Sell task, then posts a ReturnGoods task onto the worker's personal noticeboard.
  • CraftStepSystem (16, Minute): Detects CraftStep as current step. Once per game minute, accumulates progress. When accumulator passes 1.0, deducts materials and adds crafted item.

8. MoveResult — Signal-Driven Movement Completion

  1. PathfindingSystem emplaces MoveResultComponent on the character when movement reaches a terminal outcome (Arrived, Blocked, Cancelled).
  2. MoveToStepSystem hooks on_construct<MoveResultComponent> — lightweight callback that pushes the character entity into a pending queue.
  3. On the next tick, MoveToStepSystem drains the queue, reads MoveResult, and either advances to the next step (Arrived) or fails the task (Blocked/Cancelled).

MoveResult is cleared by PathfindingSystem when a new MoveGoalComponent is accepted, so stale results never interfere.

9. System Execution Order

 5  NoticeboardSystem     routes tasks to boards via on_construct/on_destroy signals
 6  CitySystem            ensures city entity has GameEntityComponent + NoticeboardComponent
11  AlchemyShopSystem     posts Hire + Sell / Craft / Gather+Deliver tasks (dynasty-owned only)
12  CarpenterShopSystem   posts Hire + Sell / Craft / Gather+Deliver tasks (dynasty-owned only)
14  TaskFailureRecoverySystem  on_destroy captures Failed Sell tasks; Update posts ReturnGoods
15  TaskClaimSystem       idle characters claim nearest eligible task
16  PathfindingSystem     runs A* for MoveGoalComponent
16  CraftStepSystem       drives CraftStep on Minute tick; produces items
17  MoveToStepSystem      drives MoveToStep; advances on MoveResult signal
18  GatherStepSystem      drives GatherStep; accumulates items
19  DeliverStepSystem     drives DeliverStep; transfers items (Tiled or GameEntity target)
19  PickupStepSystem      drives PickupStep; transfers items (Tiled or GameEntity source)
19  SellStepSystem        drives SellStep; atomic items<->gold exchange at buyer
19  DynastySystem         finalises Hire-task membership; posts new Hire tasks
20  TaskCleanupSystem     destroys Done/Failed tasks; removes CurrentTaskComponent

11. What Is Not Built Yet

  • Dynasty noticeboards — structure exists; no producers yet
  • Personal noticeboards now have one producer (TaskFailureRecoverySystem posts ReturnGoods tasks). Other planned uses: eat/sleep tasks, "drop loot at home" tasks, etc.
  • Utility scoring (trait/skill modifiers, preference weights)
  • Task timeouts or reservation expiry
  • Team tasks (multi-NPC coordination)
  • UI for decision logs (console logging only)
  • Multiple city support (currently assumes singleton city entity)
